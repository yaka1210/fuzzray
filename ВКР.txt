ВВЕДЕНИЕ

Современное программное обеспечение характеризуется постоянным ростом сложности и объёма кода, что увеличивает вероятность возникновения различных дефектов, в том числе критических уязвимостей. Этот вызов делает актуальным внедрение принципов безопасной разработки на всех этапах создания программного обеспечения. В Российской Федерации данное направление закреплено национальным стандартом ГОСТ Р 56939-2024, который устанавливает фаззинг-тестирование как обязательную часть разработки безопасного ПО [1].

Среди методов динамического тестирования фаззинг демонстрирует высокую эффективность в обнаружении ошибок, проявляющихся при специфических условиях выполнения. Однако производительность современных фаззеров создаёт новую проблему: за сутки работы инструмент может накопить сотни файлов с данными о сбоях, разбор которых вручную занимает недели. В ходе выполнения данной работы эта проблема была обозначена как «информационная перегрузка».

Помимо объёма данных, существует и качественная проблема -- «смысловой разрыв» между тем, что выдаёт фаззер (файл + номер сигнала), и тем, что нужно разработчику (тип уязвимости, место в коде, рекомендация по исправлению). Существующие инструменты анализа результатов фаззинга решают эту задачу лишь частично. Crashwalk [10] обеспечивает базовую сортировку сбоев по критичности, но не определяет конкретный тип уязвимости. FuzzManager [9] предоставляет расширенные возможности, но ориентирован на промышленное использование и требует развёртывания серверной инфраструктуры. Ни один из этих инструментов не формирует итогового отчёта с классификацией по CWE и рекомендациями по исправлению.

Цель работы -- разработка инструмента анализа результатов фаззинг-тестирования, который автоматически выполняет дедупликацию, классификацию уязвимостей по CWE, оценку их критичности и формирует структурированный HTML-отчёт.

Для достижения цели были поставлены следующие задачи:
-- изучить современные инструменты фаззинг-тестирования и форматы их выходных данных;
-- изучить существующие инструменты анализа результатов фаззинга и выявить их ограничения;
-- составить техническое задание на разработку инструмента;
-- реализовать инструмент на языке Python в виде конвейера из пяти модулей;
-- провести экспериментальную проверку на реальных выходных данных AFL++;
-- выполнить сравнительный анализ с существующими решениями.

Объект исследования -- процесс анализа результатов фаззинг-тестирования программного обеспечения.

Предмет исследования -- методы и средства автоматической классификации и приоритизации уязвимостей, обнаруженных фаззером.

Практическая значимость работы состоит в том, что разработанный инструмент может применяться разработчиками при создании программного обеспечения и специалистами по безопасности при аудите.


1 ФАЗЗИНГ-ТЕСТИРОВАНИЕ КАК МЕТОД ВЫЯВЛЕНИЯ УЯЗВИМОСТЕЙ

1.1 Методы тестирования программного обеспечения согласно ГОСТ Р 56939-2024

ГОСТ Р 56939-2024 устанавливает системный подход к разработке безопасного ПО, вводя определения ключевых видов тестирования [1]. Эти методы образуют комплекс взаимодополняющих мер, применяемых на разных этапах жизненного цикла программного обеспечения.

Статический анализ исходного кода программы -- вид работ по инструментальному исследованию программы, основанный на анализе исходных текстов программы с использованием специализированных инструментальных средств (статических анализаторов) в режиме, не предусматривающем исполнения кода. Инструменты статического анализа (SAST) сканируют код на наличие известных уязвимостей: использование небезопасных функций, риски инъекций, ошибки форматирования строк. Главными ограничениями метода являются высокий процент ложных срабатываний и невозможность обнаружить дефекты, проявляющиеся только в специфических условиях среды выполнения.

Экспертиза исходного кода программы -- вид работ по выявлению недостатков программы в исходном коде, основанный на анализе в режиме, не предусматривающем реального выполнения кода. В отличие от автоматизированного статического анализа, экспертиза предполагает привлечение специалистов для глубокого анализа логики, архитектурных решений и потоков данных. Стандарт прямо указывает, что выполнение экспертизы непосредственно создателями кода нежелательно.

Динамический анализ кода программы -- вид работ по инструментальному исследованию программы, основанный на анализе кода в режиме непосредственного исполнения. Это общий термин, охватывающий различные техники тестирования работающего приложения с целью обнаружения дефектов, проявляющихся только в процессе выполнения.

Фаззинг-тестирование программы -- вид работ по исследованию программы, направленный на оценку её свойств и основанный на передаче программе случайных или специально сформированных входных данных, отличных от данных, предусмотренных алгоритмом работы программы. Это конкретный и эффективный метод динамического анализа, предназначенный для обнаружения неизвестных уязвимостей, связанных с обработкой некорректных или неожиданных данных.

Тестирование на проникновение -- вид работ по выявлению уязвимостей программы, основанный на моделировании действий потенциального нарушителя. В отличие от фаззинга, тестирование на проникновение включает более осмысленный и сценарный подход.

Функциональное тестирование программы -- вид работ по исследованию программы, направленный на выявление отличий между реально существующими и требуемыми свойствами. В контексте безопасности оно направлено на проверку корректности реализации функций безопасности.

Композиционный анализ -- вид работ, основанный на формировании перечня зависимостей ПО, определении особенностей их использования, выявлении наличия уязвимостей в зависимостях ПО. Этот анализ помогает предотвратить проблемы безопасности, которые могут быть занесены в программу через сторонние компоненты.

Все перечисленные методы не являются изолированными. Они образуют единый процесс, где результаты одного этапа используются для планирования другого. Стандарт предписывает, что тесты для фаззинга должны разрабатываться с учётом результатов моделирования угроз, статического анализа и экспертизы исходного кода [1].

1.2 Понятие и исторический контекст фаззинг-тестирования

Фаззинг-тестирование представляет собой автоматизированную методику, при которой целевая программа подвергается интенсивной обработке специально сгенерированными или мутированными входными данными. Целью является провоцирование аномального поведения: аварийного завершения, зависания, утечки памяти или неопределённого поведения. Каждое такое событие сигнализирует о потенциальном дефекте в логике обработки данных.

Зарождение метода относится к 1988 году и связано с работой Барта Миллера «An Empirical Study of the Reliability of UNIX Utilities» [13]. Существует история, связанная с возникновением этого подхода: удар молнии вызвал искажение данных при передаче по линии связи, что привело к сбою в работе программы. Этот инцидент вдохновил профессора Миллера вместе со студентами на проведение научного исследования в данной области. Результатом их работы стала программа fuzzer. Сам Миллер определял свой метод так: «Наш подход не является заменой формальной верификации или тестирования. Скорее, это легковесный механизм для поиска ошибок в системе и повышения её надёжности» [13].

1.3 Классификация методов и инструментов фаззинга

Современный ландшафт фаззинга характеризуется разнообразием подходов. Их классификация может быть проведена по нескольким независимым осям.

По уровню информированности о тестируемом объекте:

[Здесь рисунок 1 -- Наглядная разница фаззинга по уровню информированности о тестируемом объекте]

-- black-box (чёрный ящик): фаззер не имеет доступа к исходному коду или внутренней структуре программы. Тестирование проводится исключительно через внешние интерфейсы. Подход универсален, но наименее эффективен, так как мутации носят случайный характер;
-- white-box (белый ящик): фаззер имеет полный доступ к исходному коду и может использовать методы символьного выполнения для построения точной модели программы. Максимальная эффективность, но требует значительных вычислительных ресурсов;
-- grey-box (серый ящик): компромиссный и наиболее популярный подход. Фаззер имеет частичный доступ к внутренней информации, как правило, в виде обратной связи по покрытию кода. Именно этот класс доминирует в современной практике.

По стратегии генерации входных данных:

-- мутационные: исходят из набора валидных входных данных и применяют к ним мутации: битовые перевороты, арифметические операции над байтами, удаление и вставка блоков;
-- генерационные: создают входные данные на основе правил или моделей. Позволяют генерировать сложные, семантически корректные структуры;
-- гибридные: комбинируют оба подхода.

По архитектуре исполнения:

-- in-process: код целевой функции выполняется в адресном пространстве самого фаззера. Обеспечивает максимальную скорость;
-- out-of-process: целевая программа запускается в отдельном процессе. Обеспечивает лучшую изоляцию.

По наличию обратной связи:

-- без обратной связи: фаззер генерирует данные без учёта реакции программы. Эффективность низка;
-- с обратной связью: фаззер использует информацию о покрытии кода для адаптации процесса генерации тестов. Все современные высокоэффективные фаззеры работают с обратной связью.

1.4 Ключевые механизмы современного фаззинга

Эффективность современного фаззинга основывается на трёх взаимосвязанных технологических компонентах.

Первый компонент -- обратная связь по покрытию кода. Цель фаззера -- максимизировать покрытие, заставив программу выполнить как можно больше уникальных путей. Используются метрики:
-- покрытие базовых блоков: регистрируется факт выполнения хотя бы одной инструкции внутри базового блока;
-- покрытие ветвей: регистрируется факт перехода между двумя базовыми блоками. Именно эту метрику использует AFL++ [6];
-- покрытие путей: учитывает всю последовательность выполненных базовых блоков.

Второй компонент -- санитайзеры, встраиваемые в программу на этапе компиляции [14]:
-- AddressSanitizer (ASan): обнаруживает ошибки работы с памятью -- выход за границы буфера, использование после освобождения, двойное освобождение;
-- UndefinedBehaviorSanitizer (UBSan): обнаруживает операции, приводящие к неопределённому поведению -- целочисленное переполнение, деление на ноль, некорректное преобразование типов;
-- MemorySanitizer (MSan): обнаруживает чтение неинициализированной памяти;
-- LeakSanitizer (LSan): обнаруживает утечки памяти.

Фаззер, работающий совместно с санитайзером, находит не только входные данные, приводящие к аварийному завершению, но и те, что вызывают тонкие, но критически опасные ошибки. Для программ на C/C++ это особенно важно, поскольку уязвимости вроде переполнения буфера или использования после освобождения могут не вызывать немедленного падения, но представляют серьёзную угрозу безопасности [14].

Третий компонент -- эволюционный алгоритм мутаций:

[Здесь рисунок 2 -- Процесс фаззинга как эволюционный алгоритм]

-- инициализация: создаётся начальный набор тестовых кейсов;
-- селекция: выбираются кейсы, приведшие к увеличению покрытия;
-- мутация: выбранные кейсы подвергаются мутациям (bitflip, arith, havoc);
-- исполнение и оценка: каждый мутировавший кейс запускается на целевой программе;
-- включение в корпус: если кейс привёл к новому пути, он добавляется в корпус.

Этот цикл позволяет постепенно исследовать всё более глубокие ветви логики программы, наращивая покрытие [7].

2 ПРОБЛЕМАТИКА АНАЛИЗА РЕЗУЛЬТАТОВ ФАЗЗИНГ-ТЕСТИРОВАНИЯ НА ОСНОВЕ AFL++

2.1 Организация выходных данных AFL++

Одной из ключевых проблем при работе с AFL++ является объём данных, генерируемых в процессе фаззинга. AFL++ организует результаты своей работы в чётко определённую иерархию каталогов [7, 8].

[Здесь рисунок 3 -- Структура выходного каталога AFL++]

При запуске afl-fuzz с параметром -o out_dir создаётся следующая структура:

а) каталоги с тестовыми кейсами:
-- queue/ -- содержит «интересные» входные данные, которые привели к открытию новых путей выполнения;
-- crashes/ -- включает входные данные, вызывающие аварийное завершение программы. Файлы именуются по шаблону id:XXXXXXXX,sig:NN,src:YYYYYYYY, где sig указывает на сигнал, вызвавший сбой;
-- hangs/ -- хранит тестовые кейсы, приводящие к зависанию программы;

б) файлы метаданных:
-- fuzzer_stats -- актуальная статистика выполнения, более 40 метрик;
-- plot_data -- историческая статистика в формате CSV для построения графиков;

в) служебные файлы:
-- fuzz_bitmap -- битовая карта покрытия кода;
-- cmdline -- командная строка запуска;
-- target_hash -- хеш-сумма инструментированного бинарного файла.

2.2 Основные сложности анализа

Основные сложности возникают из-за несоответствия между скоростью автоматической генерации тестовых кейсов и возможностями их ручного анализа. Современные фаззеры способны выполнять тысячи запусков в секунду, что приводит к накоплению значительного объёма данных.

Ключевая проблема -- недостаточная эффективность базовой дедупликации, которую выполняет AFL++. Инструмент использует для группировки результатов хеши путей выполнения, что позволяет отсеять только явные дубликаты. Однако на практике различные входные данные могут приводить к одной и той же уязвимости через разные пути, а AFL++ классифицирует их как отдельные сбои.

Временные затраты на ручной анализ каждого уникального сбоя составляют от 15 минут: воспроизведение, анализ под отладчиком, интерпретация результатов, оценка потенциальной эксплуатируемости. Для полноценного понимания природы сбоя недостаточно анализа только самого файла -- необходим доступ к отладочным символам и исходному коду программы.

Существенной проблемой является также высокий процент ложных и низкокритичных срабатываний. Значительная часть обнаруженных сбоев может быть связана с особенностями инструментации или некритичными дефектами, не представляющими угрозы безопасности.

2.3 Проблемы классификации и оценки критичности

После того как результаты собраны и очищены от дубликатов, встаёт главный вопрос: что именно нашёл фаззер и насколько это опасно?

Crashwalk [10] автоматизирует обработку сбоев: запускает их под отладчиком GDB и использует эвристический скрипт exploitable [16] для оценки. В результате каждый сбой получает одну из четырёх категорий: EXPLOITABLE, PROBABLY_EXPLOITABLE, UNKNOWN или NOT_EXPLOITABLE. Хотя это удобно для первичной сортировки, такая классификация остаётся поверхностной -- она не определяет конкретный тип уязвимости.

Противоположный подход -- использование каталога CWE (Common Weakness Enumeration) [11]. Однако автоматическое точное сопоставление произвольного сбоя с конкретной категорией CWE требует семантического понимания кода и контекста ошибки, что сложно для полной автоматизации.

Есть и промежуточный вариант -- классификация по сигналам операционной системы. AFL++ фиксирует сигнал, вызвавший сбой (например, SIGSEGV). Проблема в том, что один и тот же сигнал может указывать на десятки принципиально разных проблем в коде.

Возникает смысловой разрыв: с одной стороны -- быстрые, но поверхностные методы, с другой -- точные, но практически не автоматизируемые. Однако анализ промышленных данных фаззинга показывает, что подавляющее большинство опасных уязвимостей в C/C++ относится к небольшому числу классов с чёткими, автоматически распознаваемыми паттернами [11, 12]. Это открывает путь к созданию инструмента, реализующего стратегию приоритетной классификации -- фокусировку на наиболее критичных и автоматически распознаваемых типах уязвимостей.

2.4 Пример: анализ результатов фаззинга jhead-2.90

Для демонстрации описанных проблем был проведён фаззинг утилиты jhead версии 2.90, предназначенной для работы с метаданными JPEG-файлов.

[Здесь рисунок 4 -- Статус-скрин AFL++ после фаззинга утилиты jhead]

После 7 часов 24 минут работы AFL++ было получено 85 файлов с уникальными сбоями. Ручной анализ этого объёма потребовал бы следующих этапов:

-- предварительная обработка: сравнение трассировок стеков для поиска дубликатов, минимизация каждого файла утилитой afl-tmin. На 85 файлов это заняло бы 3--4 часа машинного времени;
-- анализ причин: запуск каждого файла под отладчиком GDB, определение места и причины сбоя. AFL++ сообщает только номер сигнала (например, SIGSEGV), но причина может быть любой;
-- классификация: сопоставление адреса из трассировки с функцией в исходном коде, определение типа уязвимости. На один разбор у специалиста уходит от 15 минут, на 85 сбоев -- минимум 21 час;
-- оценка и отчётность: определение критичности каждой ошибки и подготовка отчёта для разработчиков.

Время анализа в разы превышает время тестирования. Без автоматизации ценная работа фаззера остаётся лишь «папкой с битыми файлами».


3 ТЕХНИЧЕСКОЕ ЗАДАНИЕ НА РАЗРАБОТКУ ИНСТРУМЕНТА

3.1 Назначение и область применения

Инструмент FuzzRay предназначен для автоматической постобработки результатов фаззинг-тестирования, выполненного с использованием AFL++. Областью применения является процесс разработки безопасного программного обеспечения на языках C и C++.

Инструмент принимает на вход выходной каталог AFL++ и формирует структурированный HTML-отчёт, в котором каждый обнаруженный сбой классифицирован по типу уязвимости и приоритизирован по степени критичности.

3.2 Функциональные требования

Инструмент должен обеспечивать выполнение следующих функций:

1. Сбор данных: обход структуры выходного каталога AFL++, извлечение файлов сбоев из каталогов crashes/, парсинг метаданных из fuzzer_stats и plot_data, поддержка параллельного фаззинга (несколько экземпляров фаззера).

2. Трёхуровневая дедупликация: устранение точных дубликатов по хешу SHA-1 входного файла (уровень A), группировка семантически эквивалентных сбоев по нормализованному хешу стека вызовов (уровень B), объединение сбоев, указывающих на одну и ту же точку в исходном коде, в логические дефекты (уровень C).

3. Классификация: определение типа уязвимости по каталогу CWE на основе вывода санитайзеров ASan и UBSan (основной метод) и эвристического анализа через GDB (резервный метод). Перечень распознаваемых CWE: CWE-787, CWE-125, CWE-416, CWE-415, CWE-119, CWE-476, CWE-190, CWE-681, CWE-369, CWE-457, CWE-134.

4. Построение многомерной таксономии: для каждого сбоя определяются класс сигнала, место сбоя (код программы, аллокатор, строковая функция), область памяти (стек, куча, нулевая страница), состояние потока управления.

5. Оценка эксплуатируемости: присвоение каждому сбою одной из четырёх категорий: EXPLOITABLE, PROBABLY_EXPLOITABLE, PROBABLY_NOT_EXPLOITABLE, UNKNOWN.

6. Символизация: преобразование адресов точек сбоя в имена функций и расположения в исходном коде (file:line) с использованием утилиты addr2line из пакета binutils.

7. Приоритизация: вычисление числового рейтинга критичности с учётом типа CWE, оценки эксплуатируемости и частоты воспроизведения. Присвоение уровня: CRITICAL, HIGH, MEDIUM, LOW.

8. Контекстно-зависимые рекомендации: для уязвимостей с локальным синтаксическим паттерном (CWE-190, CWE-369, CWE-476, CWE-681, CWE-787, CWE-125, CWE-134) генерация рекомендации по исправлению на основе паттерн-матчинга по фрагменту исходного кода в точке сбоя с подстановкой имён реальных переменных.

9. Отображение фрагмента исходного кода: показ 3 строк до и 3 строк после точки сбоя с подсветкой строки, в которой произошёл сбой.

10. Генерация скриптов воспроизведения: создание bash-скрипта для каждого дефекта, проверяющего наличие бинарника, запускающего его с crash-файлом и интерпретирующего код возврата.

11. Минимизация входов через afl-tmin: опциональное автоматическое сокращение размера crash-файла до минимума, ещё провоцирующего сбой, с отображением hex-дампа в отчёте.

12. Генерация HTML-отчёта: формирование самодостаточного файла без внешних зависимостей, содержащего сводку, список дефектов кода с фрагментами исходного кода и рекомендациями, таблицу ошибок выполнения, график накопления сбоев.

3.3 Требования к входным и выходным данным

Входные данные:
-- путь к выходному каталогу AFL++ (обязательный);
-- путь к целевому бинарному файлу для GDB-воспроизведения (опциональный);
-- шаблон аргументов командной строки целевой программы (опциональный, по умолчанию «@@»).

Выходные данные:
-- HTML-файл отчёта, открываемый в любом современном браузере без подключения к сети.

3.4 Требования к программной среде

-- операционная система: Linux (x86-64);
-- интерпретатор: Python 3.12 или выше;
-- внешние инструменты: GDB (для режима воспроизведения сбоев), addr2line из пакета binutils (для символизации адресов), опционально afl-tmin (для минимизации входов);
-- целевой бинарный файл рекомендуется собирать с флагами `-g -fsanitize=address,undefined` -- отладочные символы необходимы для символизации, санитайзеры -- для повышения точности классификации.


4 ПРОГРАММНАЯ РЕАЛИЗАЦИЯ ИНСТРУМЕНТА FUZZRAY

4.1 Архитектура конвейера и модели данных

Инструмент FuzzRay реализован как конвейер постобработки с модульной архитектурой. Каждый модуль решает конкретную задачу и передаёт результаты следующему.

[Здесь рисунок 5 -- Блок-схема архитектуры FuzzRay. Collector -> Deduplicator -> Classifier -> Prioritizer -> Reporter]

Конвейер состоит из пяти модулей:
1. Collector -- сбор сырых данных из выходного каталога AFL++;
2. Deduplicator -- двухуровневая дедупликация сбоев;
3. Classifier -- классификация по CWE, построение таксономии, оценка эксплуатируемости;
4. Prioritizer -- вычисление рейтинга критичности;
5. Reporter -- генерация HTML-отчёта.

Центральной структурой данных является модель Crash, описанная с помощью библиотеки Pydantic v2. Она последовательно обогащается на каждом шаге конвейера. На входе модель содержит только путь к файлу и номер сигнала, а на выходе -- полный набор классификационных признаков: распределение вероятностей CWE, таксономию поведения, оценку эксплуатируемости, уровень критичности, трассировку стека и рекомендацию по исправлению.

Модель CrashRaw хранит данные, извлечённые из файловой системы: путь к файлу, имя экземпляра фаззера, номер сигнала, хеш SHA-1 содержимого, размер файла и время обнаружения.

Модель Crash расширяет эти данные полями для результатов классификации: словарь cwe_distribution хранит распределение вероятностей по классам CWE, объект taxonomy содержит многомерную таксономию поведения (класс сигнала, место сбоя, область памяти, состояние потока управления), строка exploitability -- оценку эксплуатируемости, числовые поля severity_score и severity_level -- результаты приоритизации.

Модель CrashTaxonomy представляет собой структуру из шести полей-осей: signal_class, crash_site_kind, memory_region, control_flow_state, discovery_depth, input_class. Каждая ось заполняется независимо и может принимать значение «unknown» при отсутствии достаточных данных.

Между модулями передаётся список объектов Crash. Каждый модуль -- чистая функция вида list[Crash] -> list[Crash], что обеспечивает тестируемость и простоту отладки.

4.3 Модуль сбора и трёхуровневая дедупликация данных

Модуль сбора (collector.py) отвечает за обход структуры выходного каталога AFL++ и извлечение всех необходимых данных. Модуль автоматически определяет, является ли переданный каталог каталогом отдельного экземпляра фаззера (содержит fuzzer_stats напрямую) или корневым каталогом параллельного фаззинга (содержит несколько подкаталогов). Это обеспечивает поддержку как одиночных, так и параллельных запусков AFL++.

Для каждого файла в каталоге crashes/ извлекаются: путь к файлу, номер сигнала из имени файла (парсинг регулярным выражением вида sig[:_](\d+)), хеш SHA-1 содержимого, размер и время модификации. Из файла fuzzer_stats парсятся метрики: количество выполнений, скорость, число уникальных сбоев, командная строка запуска. Из файла plot_data парсятся точки временного ряда в формате CSV для построения графика накопления сбоев.

Результатом работы модуля является кортеж из трёх списков: CrashRaw (сырые данные сбоев), FuzzerStats (статистика фаззера) и PlotPoint (точки временного графика).

Модуль дедупликации (deduplicator.py) реализует трёхуровневую стратегию. Уровни применяются последовательно, каждый следующий работает с результатами предыдущего и сжимает множество ещё сильнее. Файлы, отнесённые к одной группе на любом из уровней, не теряются -- их пути сохраняются в списке duplicate_paths представителя группы, а счётчик duplicate_count показывает общее число триггеров.

Уровень A -- дедупликация по хешу содержимого. При параллельном фаззинге несколько экземпляров AFL++ могут сохранить идентичные файлы. Группировка по SHA-1 хешу позволяет устранить точные дубликаты. Применяется до классификации, поскольку хеш содержимого не зависит от результатов GDB-воспроизведения.

Уровень B -- дедупликация по стеку вызовов. Разные входные файлы могут приводить к одному и тому же дефекту в коде, но через разные пути выполнения. Для объединения таких случаев используется нормализованный хеш стека вызовов.

Нормализация включает: удаление адресов (замена 0x... на пустую строку), отбрасывание смещений после символа «+», фильтрацию «шумовых» фреймов -- внутренних функций санитайзеров (__asan_, __ubsan_), системных вызовов (__pthread_kill, __GI_raise, __GI_abort) и точки входа (_start). После нормализации берутся первые 5 значимых фреймов и вычисляется SHA-1 их конкатенации. Сбои с одинаковым хешем стека и одинаковым top CWE объединяются в одну группу.

Уровень C -- дедупликация по точке сбоя в исходном коде. Уровень B оставляет в отчёте все сбои, отличающиеся хотя бы одним фреймом стека -- даже если они указывают на одну и ту же строку кода через разные пути выполнения. Это технически корректно с точки зрения программной инженерии, но для разработчика, исправляющего код, такие сбои представляют один и тот же дефект. Уровень C объединяет их по ключу (file:line, инструкция в точке сбоя, top CWE). На практике этот уровень даёт наиболее значимое сокращение объёма отчёта: на утилите jhead-2.90 уровень B оставляет 63 уникальные группы по стеку, а уровень C сжимает их до 8 уникальных дефектов кода.

4.4 Модуль классификации уязвимостей и символизация

Модуль классификации -- наиболее сложная часть инструмента. Он реализован как пакет classifier/ и выполняет три задачи: определение типа уязвимости (CWE), построение таксономии поведения и оценку эксплуатируемости.

Определение CWE.

Классификатор возвращает не жёсткую метку, а распределение вероятностей: словарь вида {CWE-787: 0.85, CWE-125: 0.10, unknown: 0.05}. Значения складываются из весов правил, где каждый источник данных имеет свой уровень достоверности. Итоговое распределение нормализуется так, чтобы сумма равнялась единице.

Источники данных используются в следующем порядке приоритета:

1. Сигнал из имени файла AFL++. Номер сигнала извлекается из имени crash-файла и даёт начальное априорное распределение. Например, SIGSEGV (сигнал 11) даёт равномерную вероятность между CWE-787, CWE-125 и CWE-476 с весами порядка 0.25. SIGFPE (сигнал 8) даёт CWE-369 с весом 0.7. Эти веса низкие, потому что один сигнал соответствует десяткам разных уязвимостей.

2. Вывод санитайзеров. Наиболее надёжный источник. Санитайзер инструментировал код и точно знает, что произошло. Модуль sanitizer.py парсит текстовый вывод программы регулярными выражениями. Например, сообщение ASan «heap-buffer-overflow» в сочетании с «WRITE of size» однозначно указывает на CWE-787 с весом 0.95. Сообщение «heap-use-after-free» указывает на CWE-416 с весом 0.97.

[Здесь таблица 3 -- Распознаваемые CWE и методы обнаружения. Столбцы: CWE-ID, Название, Основной метод (санитайзер), Резервный метод (GDB), Вес. Строки: CWE-787 Out-of-Bounds Write, CWE-125 Out-of-Bounds Read, CWE-416 Use After Free, CWE-415 Double Free, CWE-119 Buffer Operations, CWE-476 NULL Pointer Dereference, CWE-190 Integer Overflow, CWE-681 Incorrect Type Conversion, CWE-369 Divide By Zero, CWE-457 Uninitialized Variable, CWE-134 Format String]

Выбор 11 классов CWE построен на двух уровнях: базовом (6 CWE) и расширенном (+5 CWE).

Базовые шесть взяты из пересечения двух авторитетных источников: рейтинга MITRE CWE Top 25 Most Dangerous Software Weaknesses 2024 [11] и отраслевого отчёта Code Intelligence «Most Dangerous CWEs in C 2025» [12]. В пересечение входят CWE-787 (Out-of-Bounds Write), CWE-125 (Out-of-Bounds Read), CWE-416 (Use After Free), CWE-119 (Improper Restriction of Operations within Buffer), CWE-476 (NULL Pointer Dereference), CWE-190 (Integer Overflow). Это уязвимости, которые фаззеры реально находят чаще всего и которые имеют наибольший потенциал эксплуатации. Дополнительным эмпирическим подтверждением служат публичные отчёты Google OSS-Fuzz [15], где heap-buffer-overflow и use-after-free суммарно составляют более 60 процентов находок.

Расширенные пять добавлены каждый со своим самостоятельным обоснованием:

-- CWE-415 (Double Free). Включён, потому что glibc в современных версиях обнаруживает двойное освобождение и завершает программу через abort с диагностическим сообщением «double free detected», а ASan генерирует отдельный класс ошибки double-free. Это даёт чёткий машинно-распознаваемый паттерн. Обоснование: оригинальная статья ASan [14] описывает double-free как один из шести базовых классов ошибок памяти.

-- CWE-369 (Divide By Zero). Включён, потому что любая программа на C, выполняющая целочисленное деление, потенциально подвержена этой уязвимости, а её обнаружение тривиально: SIGFPE с инструкцией div или idiv в точке сбоя или сообщение UBSan integer-divide-by-zero.

-- CWE-457 (Use of Uninitialized Variable). Включён, потому что MemorySanitizer (MSan) специально создан для обнаружения этого класса ошибок и выдаёт стабильный текстовый паттерн use-of-uninitialized-value. Обоснование: статья Stepanov & Serebryany «MemorySanitizer: fast detector of uninitialized memory use in C++» [31] -- основополагающая работа, доказывающая практичность автоматического обнаружения этого класса.

-- CWE-681 (Incorrect Conversion between Numeric Types). Включён по результатам экспериментальной проверки на jhead-2.90 (см. главу 5): при анализе результатов фаззинга было обнаружено, что значительная часть UBSan-трапов соответствует обработчикам FloatCastOverflow (handler 5) и ImplicitConversion (handler 7), которые семантически точнее описываются как некорректное преобразование типов, а не общее целочисленное переполнение. Обоснование: исходный код LLVM, файл clang/lib/CodeGen/CodeGenFunction.h [26], где эти обработчики определены отдельно от обработчиков переполнения арифметики.

-- CWE-134 (Format String). Включён, потому что компиляторы GCC и Clang выдают предупреждение -Wformat-security при обнаружении пользовательских данных в форматной строке, а сбой при эксплуатации проявляется крахом внутри vfprintf или vsnprintf, что распознаётся через анализ верхнего фрейма стека.

В итоге набор из 11 классов покрывает все распространённые типы уязвимостей памяти и неопределённого поведения в C/C++, при этом каждый класс имеет независимый машинно-распознаваемый паттерн в выходе санитайзеров или GDB.

Символизация адресов через addr2line.

После сбора данных от санитайзера и GDB модуль классификации выполняет символизацию: преобразование адресов точек сбоя в имена функций и расположения в исходном коде. Для этого используется утилита addr2line из пакета GNU Binutils [33], которая по адресу инструкции и пути к бинарному файлу с отладочными символами возвращает имя функции и пару file:line.

Реализация символизации сосредоточена в подмодуле classifier/symbolizer.py. Для каждого сбоя символизации подвергаются: программный счётчик ($pc) -- адрес инструкции, на которой произошёл сбой; все адреса фреймов в стеке вызовов -- для обогащения backtrace расположениями в исходном коде. Результат сохраняется в полях crash_function и crash_location модели Crash.

Для случаев, когда $pc указывает в код стандартной библиотеки (libc, nptl, sysdeps) -- что типично для сбоев, инициированных через UBSan trap или abort() -- символизация $pc возвращает путь внутри libc, который не интересен пользователю. В этих случаях модуль выполняет fallback-поиск: проходит по фреймам стека сверху вниз и берёт первый осмысленный фрейм, относящийся к коду пользовательской программы. Шумовые фреймы (__pthread_kill, __GI_raise, __GI_abort, __libc_start_main) и фреймы из системных каталогов библиотек игнорируются.

Символизация даёт два важных эффекта. Во-первых, рекомендации по исправлению указывают конкретную строку кода, а не только имя функции, что значительно снижает время локализации дефекта разработчиком. Во-вторых, наличие пары (file, line) служит ключом для уровня C дедупликации и даёт возможность отображать в отчёте фрагмент исходного кода в точке сбоя.

3. GDB-воспроизведение. При отсутствии вывода санитайзеров включается резервный механизм. Модуль gdb_runner.py запускает целевую программу с crash-файлом в GDB batch-режиме и собирает: трассировку стека (bt 20), значения регистров (info registers), инструкцию в точке сбоя (x/1i $pc), адрес обращения (si_addr). Эвристики анализируют собранные данные:

-- если адрес обращения меньше 0x1000 -- вероятен NULL pointer dereference (CWE-476, вес 0.75);
-- если инструкция в точке сбоя содержит div/idiv -- вероятно деление на ноль (CWE-369, вес 0.8);
-- если инструкция -- mov с записью по адресу -- вероятна запись за границами буфера (CWE-787, вес 0.6);
-- если в стеке вызовов присутствуют функции аллокатора (malloc, free, realloc) -- вероятен use-after-free или повреждение кучи (CWE-416, вес 0.5);
-- если в стеке присутствуют строковые функции (memcpy, strcpy, strlen) -- вероятен выход за границы буфера (CWE-787, вес 0.6);
-- если в сыром выводе GDB обнаружена строка «double free detected» -- вероятно двойное освобождение (CWE-415, вес 0.85);
-- если в сыром выводе обнаружена строка «corrupted size vs. prev_size» или «invalid next size» -- вероятно повреждение метаданных кучи (CWE-787, вес 0.7).

Отдельная эвристика обрабатывает случай UBSan в trap-режиме. Когда UBSan компилирует код с флагом -fsanitize-trap (поведение по умолчанию при совместном использовании с AFL++), вместо текстового сообщения он вставляет инструкцию ud1 с закодированным номером обработчика. Инструмент декодирует этот номер и сопоставляет его с CWE по таблице из файла SanitizerHandler.h компилятора Clang [26].

[Здесь таблица 4 -- Маппинг UBSan handler -> CWE. Столбцы: Номер, Имя обработчика, CWE, Вес. Примеры строк: 0 AddOverflow CWE-190 0.75, 3 DivremOverflow CWE-369 0.8, 5 FloatCastOverflow CWE-681 0.7, 7 ImplicitConversion CWE-681 0.65, и т.д.]

Веса эвристик были откалиброваны эмпирически исходя из надёжности каждого источника информации: прямое сообщение санитайзера получает 0.95, декодирование UBSan handler -- 0.7--0.8, эвристика по инструкции -- 0.5--0.6, априорное распределение по сигналу -- 0.2--0.3.

Дополнительно реализовано обнаружение переполнения стека вызовов (stack exhaustion): если в первых 10 фреймах стека повторяется не более двух уникальных функций, сбой классифицируется как бесконечная рекурсия, и все эвристики CWE сбрасываются, чтобы не давать ложных срабатываний.

Параллельное воспроизведение. GDB-воспроизведение каждого сбоя занимает от 1 до 30 секунд. При десятках сбоев это становится узким местом. Поскольку воспроизведения независимы друг от друга, они были распараллелены с использованием concurrent.futures.ProcessPoolExecutor из стандартной библиотеки Python. Количество рабочих процессов по умолчанию равно числу ядер процессора, но может быть задано пользователем через флаг --jobs.

Таксономия поведения.

Помимо определения CWE, для каждого сбоя строится многомерная таксономия поведения из четырёх осей:

-- signal_class: класс сигнала операционной системы. ERROR_SEGFAULT (SIGSEGV), ERROR_ABORT (SIGABRT), ERROR_FPE (SIGFPE), ERROR_BUSERROR (SIGBUS), ERROR_ILL (SIGILL), ERROR_TIMEOUT, ERROR_UNKNOWN;

-- crash_site_kind: где именно произошёл сбой, определяется по верхнему фрейму стека вызовов. user_code (код программы), libc_alloc (внутри malloc/free -- указывает на повреждение кучи выше по стеку), libc_string (внутри memcpy/strcpy -- указывает на передачу буфера недостаточного размера), libc_io (внутри printf/fwrite), dynamic_linker, kernel_syscall;

-- memory_region: область памяти обращения, определяется по адресу si_addr. null_page (адрес меньше 0x1000), heap, stack, bss, mmap, kernel_space (адрес выше 0xFFFF800000000000);

-- control_flow_state: состояние потока управления, определяется по значению $pc. normal ($pc в допустимом диапазоне), ret_to_unmapped ($pc меньше 0x1000 -- возврат в невалидный адрес), stack_exhaustion (длина стека более 200 фреймов).

Эти оси заполняются независимо, любая может быть «unknown». В отчёте это даёт матричную группировку: вместо плоского «85 SIGSEGV» получается «SIGSEGV * libc_string * heap = 9 (OOB на memcpy); SIGSEGV * user_code * null_page = 4 (NULL deref в парсере)».

Оценка эксплуатируемости.

Модуль exploitability.py реализует балльную систему оценки, вдохновлённую подходом CERT exploitable [16], но адаптированную для работы с результатами собственного классификатора.

Итоговый балл складывается из трёх компонентов:

1. Балл по типу CWE: CWE-787 и CWE-416 получают 6 баллов (запись в произвольный адрес или контроль через dangling pointer -- прямой путь к эксплуатации). CWE-415 и CWE-134 -- 5 баллов. CWE-125 -- 2 балла (чтение за границами -- утечка данных, но не выполнение кода). CWE-476 и CWE-369 -- 1 балл (обычно только отказ в обслуживании).

2. Балл по таксономии: возврат в невалидный адрес (ret_to_unmapped) добавляет 4 балла, повреждённый косвенный вызов -- 5 баллов, сбой в аллокаторе -- 2 балла, null_page вычитает 1 балл.

3. Балл по данным GDB: если faulting address выше 0x1000 при SIGSEGV -- +1, если инструкция call/jmp при SIGSEGV -- +3, если $pc ниже 0x1000 -- +4.

Пороги: 8 и выше -- EXPLOITABLE, 5--7 -- PROBABLY_EXPLOITABLE, 2--4 -- PROBABLY_NOT_EXPLOITABLE, менее 2 -- UNKNOWN.

4.5 Модуль приоритизации и генерации отчёта

Приоритизация.

Модуль prioritizer.py вычисляет для каждого сбоя числовой рейтинг критичности и присваивает текстовый уровень.

Формула:

severity_score = cwe_base * confidence_penalty * exploit_mult + freq_bonus

где:

-- cwe_base -- базовый балл по типу CWE. CWE-787 и CWE-416 получают 9 (критические уязвимости записи), CWE-415 -- 8, CWE-125, CWE-119, CWE-134 -- 7, CWE-476, CWE-190, CWE-681, CWE-369, CWE-457 -- 5;

-- confidence_penalty = 1 - (1 - confidence) * 0.5, где confidence -- максимальное значение в распределении CWE. Если уверенность классификации низкая, итоговый балл снижается;

-- exploit_mult -- множитель эксплуатируемости: EXPLOITABLE -- 1.5, PROBABLY_EXPLOITABLE -- 1.2, PROBABLY_NOT_EXPLOITABLE -- 0.8, UNKNOWN -- 1.0;

-- freq_bonus = log2(duplicate_count + 1) -- бонус за частоту: сбой, в который упирается большая часть корпуса, приоритетнее единичного.

Уровни критичности: CRITICAL (>= 9), HIGH (7--8), MEDIUM (5--6), LOW (< 5).

Для сбоев, которым не удалось присвоить CWE (top_cwe = «unknown»), уровень определяется по классу сигнала: ERROR_SEGFAULT, ERROR_ABORT, ERROR_BUSERROR -- HIGH, ERROR_ILL, ERROR_FPE -- MEDIUM, ERROR_TIMEOUT, ERROR_UNKNOWN -- LOW.

Генерация отчёта.

Модуль reporter/ формирует самодостаточный HTML-файл: CSS встроен в тег <style>, JavaScript -- в <script>, внешние ресурсы не подключаются. Отчёт корректно открывается в любом современном браузере без подключения к сети.

Шаблон отчёта написан на языке Jinja2 и содержит следующие секции:

1. Заголовок: имя целевого бинарника, дата генерации отчёта.

2. Сводка: четыре карточки с метриками -- общее число файлов сбоев, число уникальных уязвимостей после дедупликации, число критических и серьёзных.

3. График накопления сбоев: inline SVG, генерируемый модулем svg_chart.py по данным plot_data. Отображает количество уникальных сбоев во времени. Оформлен в тёмной цветовой схеме с градиентной заливкой и подсветкой точек данных.

4. Уязвимости (CWE): приоритизированный список дефектов кода. Каждая карточка содержит CWE-ID с названием, уровень критичности (цветовая индикация: красный -- CRITICAL, оранжевый -- HIGH, жёлтый -- MEDIUM, зелёный -- LOW), оценку эксплуатируемости, точку сбоя в виде функция (file:line), фрагмент исходного кода с подсветкой строки, в которой произошёл сбой, трассировку стека, инструкцию в точке сбоя, рекомендацию по исправлению, ссылку на каталог MITRE для подробного описания типа уязвимости. Если карточка соответствует нескольким триггерам (после уровня C дедупликации), отображается счётчик и сворачиваемый список всех файлов-триггеров.

5. Ошибки выполнения (ERROR): сбои, которым не удалось присвоить CWE. Группируются по классу сигнала.

6. Сводные таблицы: распределение по классам сигналов, местам сбоя, областям памяти, оценкам эксплуатируемости.

Фрагмент исходного кода в точке сбоя.

Подмодуль classifier/source_snippet.py читает исходный файл по полученному в результате символизации пути и извлекает 3 строки до и 3 строки после строки сбоя. Если crash_location указывает на относительный путь (например, exif.c:584), модуль выполняет поиск файла в нескольких кандидатных каталогах: рядом с целевым бинарником, в его родительском каталоге, в текущем рабочем каталоге. Это позволяет находить исходные файлы даже когда таргет был собран в одной директории, а запуск осуществляется из другой. На jhead-2.90 фрагмент исходного кода был успешно извлечён для 62 из 63 уязвимостей уровня B.

Контекстно-зависимые рекомендации.

Подмодуль classifier/recommender.py реализует генерацию рекомендаций на основе паттерн-матчинга по фрагменту исходного кода. Для каждого из 7 классов CWE с локальным синтаксическим паттерном (CWE-190, CWE-369, CWE-476, CWE-681, CWE-787, CWE-125, CWE-134) определён набор регулярных выражений, который сканирует строку сбоя и возвращает рекомендацию с подстановкой имён реальных переменных из этого кода.

Например, для CWE-190 паттерн `(\w+)\s*\*\s*(\w+)` обнаруживает умножение `len * count` и возвращает рекомендацию: «Умножение `len * count` может переполниться. Используйте проверяемую арифметику: `if (__builtin_mul_overflow(len, count, &result)) return ERR;`». Для CWE-476 паттерн `(\w+)\s*->\s*(\w+)` обнаруживает разыменование `ptr->field` и возвращает рекомендацию: «Разыменование `ptr->field` без проверки на NULL. Добавьте перед этой строкой: `if (ptr == NULL) return ERR;`».

Для классов уязвимостей, требующих анализа потока выполнения (CWE-416 Use After Free, CWE-415 Double Free, CWE-457 Uninitialized Variable, CWE-119 Generic Buffer Issue), контекстная рекомендация не генерируется -- регулярный паттерн в одной строке кода не способен дать корректное предложение исправления для этих классов. В таких случаях используется статическая рекомендация общего вида и приводится ссылка на каталог MITRE для подробного описания.

Если паттерн-матчинг не нашёл совпадений для класса, поддерживающего контекстную рекомендацию, инструмент также ложится на статический шаблон. Это происходит, например, когда крэш-линия не содержит явной арифметики, но классификатор отнёс сбой к CWE-190 (что возможно при инлайнинге UBSan или сдвиге места UB-поведения относительно отображаемой строки). Такое поведение является честной границей возможностей паттерн-матчингового подхода: лучше дать общий совет, чем сгенерировать ложную конкретную рекомендацию.

Скрипты воспроизведения.

Подмодуль reporter/reproducer.py генерирует bash-скрипт для каждого дефекта кода. Скрипт принимает путь к целевому бинарнику и crash-файлу, проверяет их наличие, запускает целевую программу с crash-файлом и интерпретирует код возврата. Скрипты сохраняются в подкаталоге <output>_reproducers/ со структурированными именами вида reproduce_001_CWE_190.sh.

Это даёт разработчику возможность мгновенно воспроизвести любой сбой одной командой, без необходимости вручную составлять командную строку с правильным crash-файлом из десятков, имеющихся в выходном каталоге AFL++.

Минимизация входных данных.

Подмодуль classifier/minimizer.py реализует опциональную интеграцию с утилитой afl-tmin из пакета AFL++ [6]. Утилита выполняет последовательное сокращение размера crash-файла -- удаление байтов, замену блоков на нулевые значения -- с проверкой того, что упрощённый файл по-прежнему вызывает тот же сбой. Результатом является минимальный воспроизводящий вход, который может быть в десятки раз меньше исходного.

Минимизация активируется флагом командной строки --minimize. Минимизированные файлы сохраняются в подкаталоге <output>_minimized/, в HTML-отчёт добавляется секция с hex-дампом первых 256 байт минимизированного входа и коэффициентом сжатия. Минимизация требует наличия afl-tmin в PATH; если утилита недоступна, флаг игнорируется без ошибки.

4.6 Интерфейс командной строки

CLI реализован с использованием библиотеки Typer. Все параметры типизированы через аннотации, справка --help генерируется автоматически.

Основные параметры:
-- --afl-out -- путь к выходному каталогу AFL++ (обязательный);
-- --target -- путь к целевому бинарнику для GDB-воспроизведения;
-- --target-args -- шаблон аргументов, @@ заменяется на путь к crash-файлу;
-- -o / --output -- путь к выходному HTML-файлу;
-- --no-replay -- режим без GDB-воспроизведения (только статическая классификация);
-- -j / --jobs -- число параллельных GDB-воспроизведений;
-- --minimize -- запуск afl-tmin для минимизации каждого crash-входа;
-- --no-reproducers -- отключение генерации bash-скриптов воспроизведения.

Примеры запуска:

  uv run fuzzray --afl-out ./out --target ./target -o report.html

  uv run fuzzray --afl-out ./out --no-replay -o report.html

  uv run fuzzray --afl-out ./out --target ./target -j 8 -o report.html

Модуль pipeline.py оркестрирует выполнение конвейера: последовательно вызывает collect, deduplicate (уровень A), classify (включая символизацию), deduplicate_by_stack (уровень B), deduplicate_by_location (уровень C), prioritize, опциональный minimize для каждого дефекта, генерацию скриптов воспроизведения и финальный render_html. Промежуточные результаты выводятся в терминал с помощью Rich: количество собранных файлов, количество уникальных после каждого из трёх уровней дедупликации, распределение по уровням критичности.

6 СРАВНИТЕЛЬНЫЙ АНАЛИЗ С СУЩЕСТВУЮЩИМИ РЕШЕНИЯМИ

6.1 Crashwalk и CERT exploitable

Crashwalk [10] -- инструмент на языке Go, автоматизирующий обработку crash-файлов. Он запускает каждый сбой под GDB с подключённым плагином CERT exploitable [16] и группирует результаты.

CERT exploitable -- плагин GDB, разработанный координационным центром CERT при Университете Карнеги-Меллона. Он анализирует состояние регистров и стек вызовов в момент сбоя и присваивает одну из четырёх категорий: EXPLOITABLE, PROBABLY_EXPLOITABLE, UNKNOWN, NOT_EXPLOITABLE.

Ключевые ограничения связки Crashwalk + exploitable:

1. Отсутствие классификации по CWE. Инструмент сообщает только уровень эксплуатируемости, но не определяет тип уязвимости. Разработчик видит «EXPLOITABLE», но не знает, что это -- переполнение буфера, use-after-free или format string.

2. Некорректная работа с UBSan в trap-режиме. В ходе выполнения данной работы было обнаружено, что CERT exploitable классифицирует все сбои, вызванные UBSan в trap-режиме (инструкция ud1), как EXPLOITABLE или PossibleStackCorruption. Это ложноположительные срабатывания: реальная причина сбоя -- целочисленное переполнение или некорректное преобразование типов, а не повреждение стека. Плагин не понимает механизм UBSan trap и интерпретирует инструкцию ud1 как признак эксплуатируемости.

3. Отсутствие рекомендаций по исправлению. Разработчик должен самостоятельно определить причину и способ устранения.

4. Отсутствие итогового отчёта. Результаты выводятся в терминал построчно, без визуальной группировки и приоритизации.

6.2 FuzzManager

FuzzManager [9] -- платформа для управления фаззингом, разработанная Mozilla. Она состоит из серверного компонента (Django-приложение с базой данных) и клиентских агентов, отправляющих результаты на сервер.

Возможности:
-- хранение и группировка crash-файлов по сигнатурам стеков;
-- веб-интерфейс для просмотра и фильтрации;
-- интеграция с Bugzilla для автоматического создания баг-репортов;
-- поддержка множества фаззеров и целей.

Ключевые ограничения:

1. Сложность развёртывания. Требуется установка серверного компонента (Django + PostgreSQL), настройка агентов, конфигурация сетевого взаимодействия. Для одиночного использования или небольшого проекта это избыточно.

2. Ориентация на промышленное использование. Инструмент спроектирован для организаций, ведущих непрерывный фаззинг десятков целей. Для анализа результатов одного запуска AFL++ он нецелесообразен.

3. Отсутствие классификации по CWE. FuzzManager группирует сбои по сигнатурам стеков, но не определяет тип уязвимости.

4. Отсутствие самодостаточного отчёта. Результаты доступны только через веб-интерфейс сервера.

Crashwalk даёт только оценку эксплуатируемости без типизации уязвимости. FuzzManager предоставляет управление фаззингом, но требует инфраструктуры и не выполняет содержательной классификации.


ЗАКЛЮЧЕНИЕ

В ходе выполнения данной работы был разработан инструмент FuzzRay для автоматического анализа результатов фаззинг-тестирования на основе AFL++.

Были решены все поставленные задачи:

1. Изучены современные инструменты фаззинг-тестирования и форматы их выходных данных. Выбран AFL++ как наиболее универсальный и распространённый инструмент с удобной для обработки структурой выходных данных.

2. Изучены существующие инструменты анализа результатов фаззинга -- Crashwalk, CERT exploitable и FuzzManager. Выявлены их ключевые ограничения: отсутствие классификации по CWE, некорректная работа с UBSan в trap-режиме, отсутствие итогового отчёта с рекомендациями.

3. Составлено техническое задание на разработку инструмента с описанием функциональных требований, форматов входных и выходных данных, требований к программной среде.

4. Реализован инструмент на языке Python в виде конвейера из пяти модулей: сбор данных, трёхуровневая дедупликация (по SHA-1 содержимого, по нормализованному хешу стека вызовов, по точке сбоя в исходном коде), классификация (11 классов CWE + таксономия из 4 осей + оценка эксплуатируемости + символизация через addr2line), приоритизация и генерация HTML-отчёта с фрагментами исходного кода, контекстно-зависимыми рекомендациями и скриптами воспроизведения.

5. Проведена экспериментальная проверка на реальных данных. На утилите jhead-2.90 инструмент за 3 минуты обработал 85 файлов сбоев и выделил 8 уникальных дефектов кода четырёх типов CWE с точностью классификации 100% и итоговым коэффициентом сжатия 10.6. Идентичные результаты повторного запуска (14 часов фаззинга вместо 7) подтвердили воспроизводимость классификации.

6. Выполнен сравнительный анализ с существующими решениями. FuzzRay является единственным из рассмотренных инструментов, сочетающим классификацию по CWE, рекомендации по исправлению и самодостаточный HTML-отчёт.

Практическая значимость работы подтверждена экспериментально: инструмент сокращает время анализа результатов фаззинга с часов ручной работы до минут автоматической обработки, при этом предоставляя разработчику информацию не только о факте сбоя, но и о типе уязвимости, её критичности и способе исправления.
